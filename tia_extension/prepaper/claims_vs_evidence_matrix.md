# Claims Vs Evidence Matrix

This matrix defines conservative manuscript language for the current TIA extension evidence package.

| Claim | Evidence available | Evidence strength | Allowed wording | Forbidden wording | Remaining gap |
|---|---|---|---|---|---|
| The MATLAB behavioural model can map design regions quickly. | Reproducible MATLAB scripts, baseline response, full sweep CSVs, design-region maps. | Moderate for pre-design screening. | "The MATLAB behavioural workflow maps project-defined design regions across selected `Rf`, `Cpd`, `Cf`, and `ft_Hz` assumptions." | "Complete design automation"; "universal design rule"; "guaranteed stable." | Literature positioning and sensitivity discussion. |
| Increasing `Cf` generally reduces bandwidth and peaking risk. | Bandwidth versus `Cf`, peaking versus `Cf`, representative responses. | Strong within the explored behavioural sweep. | "In the selected behavioural sweep, increasing `Cf` reduces bandwidth and generally lowers peaking." | "Always stable"; "guaranteed stable"; "universal compensation rule." | Explain parameter-range dependence. |
| Design-region maps organize Safe, Marginal, and Risky operating regions. | Design-region map, safe-window fraction map, classification helper output. | Moderate as a project-defined screening method. | "Safe, Marginal, and Risky are project-defined labels for organizing extracted behavioural metrics." | "Certified safe"; "hardware stable"; "industry-standard region." | Final threshold justification and expert review. |
| Vendor SPICE macromodels show different bandwidth and peaking boundaries. | OP27 smoke test, OPA818 Cf sweep, ADA4817 Cf sweep, combined vendor summary. | Moderate to strong as macromodel evidence. | "The three vendor macromodel sets show model-dependent bandwidth and peaking trends." | "Complete validation"; "hardware validated"; "experimentally validated." | Broader model set and hardware work if required. |
| OP27, OPA818, and ADA4817 provide three real vendor macromodel comparison sets. | `spice_comparison_summary_vendor_models.csv`, individual import reports, processed CSVs. | Strong for repository evidence traceability. | "The repository includes three real vendor SPICE macromodel comparison sets: OP27, OPA818, and ADA4817." | "Three hardware prototypes"; "complete SPICE coverage." | Optional additional vendors if scope expands. |
| OP27 highlights low-GBW / small-`Cf` peaking risk. | OP27 risky, safe-candidate, and safe cases plus OP27 sweep figure. | Moderate as a smoke-test comparison. | "The OP27 smoke test illustrates strong peaking in a small-`Cf` case." | "OP27 is unsuitable in all TIAs"; "OP27 result proves hardware instability." | Clarify OP27 role as smoke test. |
| OPA818 provides high-speed vendor macromodel evidence. | Four real OPA818 LTspice cases and summary CSV. | Moderate as vendor macromodel evidence. | "The selected OPA818 vendor macromodel sweep shows little peaking over the tested `Cf` range." | "OPA818 is proven optimal"; "hardware validated." | Device-specific operating constraints and layout review. |
| ADA4817 adds a third vendor case with a borderline small-`Cf` result. | Four real ADA4817 LTspice cases and summary CSV. | Moderate as vendor macromodel evidence. | "The ADA4817 sweep adds a third vendor set and shows a borderline `Cf = 0.5 pF` case." | "ADA4817 is guaranteed stable"; "experimentally validated." | Review model assumptions and operating-point details. |
| Noise analysis provides first-pass trade-off estimates. | Noise baseline, noise trade-off CSVs, noise figures. | Moderate for relative behavioural estimates. | "The noise workflow provides first-pass behavioural estimates under stated assumptions." | "Measured noise"; "measured noise validated"; "detector noise experimentally confirmed." | SPICE noise or measured noise if required by scope. |
| The workflow is reproducible. | Scripts, CSVs, figure manifest, change logs, markdown reports. | Strong for repository reproducibility. | "The figures and tables are traceable to committed scripts and CSV source data." | "Reproduces hardware measurements"; "complete validation." | External environment and MATLAB version documentation. |
| The workflow is not hardware validated. | Project limitations and absence of hardware data. | Strong as a boundary statement. | "No hardware measurement or hardware validation is claimed." | "Hardware validated"; "bench validated"; "experimentally validated." | Future prototype measurements if needed. |
| The workflow is not measured-noise validated. | Noise assumptions and noise reports state behavioural estimates only. | Strong as a boundary statement. | "Noise results are first-pass behavioural estimates, not measured noise." | "Measured noise"; "measured noise validated"; "experimental noise evidence." | Future measured detector or SPICE noise evidence if needed. |
| The current package is a Q3 pre-paper prototype, not a final submission-ready manuscript. | Q3 readiness report, Round 10 skeleton, figure/table plan, claims matrix. | Strong for project status. | "The package is a Q3 pre-paper prototype with manuscript planning artifacts." | "Journal-ready"; "Q3 accepted"; "final submission-ready"; "complete validation." | Related work, full manuscript drafting, captions, target selection, and domain review. |
